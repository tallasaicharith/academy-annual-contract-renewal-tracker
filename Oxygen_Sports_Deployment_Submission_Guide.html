🚀 Deployment Guide ✅ Submission Checklist
Part 1

Deploy the App — Step by Step

Frontend on Vercel, backend + database on Render. Do these in order — the backend and database need to exist before the frontend has anything to talk to.

Read this before you start If your backend is currently using a SQLite file for the database, it will NOT survive on Render's free tier — the disk resets on every redeploy and every seeded user/contract disappears. Step 1 below moves you to a real, persistent Postgres database first, so this never bites you after go-live.
STEP 1

Push your code to GitHub

Both Vercel and Render deploy by connecting directly to a GitHub repository — make sure your latest frontend and backend code (in their own folders, e.g. /frontend and /backend) is committed and pushed before continuing.

STEP 2

Create the database (Render PostgreSQL)

  1. Go to render.com → New → PostgreSQL
  2. Name it something clear, e.g. oxygen-sports-db; pick the free tier and a region close to Hyderabad
  3. Once it's created, copy the Internal Database URL shown on its page — keep that tab open, you need it in Step 3
STEP 3

Deploy the backend (Render Web Service)

  1. Render → New → Web Service → connect your GitHub repo
  2. Root Directory: backend (skip this if your backend isn't in a subfolder)
  3. Build Command: npm install
  4. Start Command: node server.js — match this to whatever your package.json actually runs
  5. Add the environment variables below (Render → your service → Environment)
VariableValue
DATABASE_URLThe Internal Database URL from Step 2
JWT_SECRETA long random string you generate yourself — never reuse an example from a tutorial
PORT5000 (Render usually sets this automatically, but set it explicitly if your code reads it)

Once it's live:

  1. Copy the public backend URL Render gives you, e.g. https://oxygen-sports-backend.onrender.com — you'll need it in Step 4
  2. Open the Shell tab on your Render service and run your seed script once, e.g. node database.js — this creates the tables and the 3 demo users
STEP 4

Deploy the frontend (Vercel)

  1. vercel.com → Add New → Project → import the same GitHub repo
  2. Root Directory: frontend (skip if not in a subfolder)
  3. Framework Preset: Vite (Vercel usually detects this automatically) — Build Command npm run build, Output Directory dist
  4. Add environment variable VITE_API_URL = the backend URL you copied in Step 3
  5. Click Deploy
Avoid the preview-link trap Use the deployment marked Production in Vercel → Deployments — not a branch preview URL containing -git-branchname-. Preview URLs are gated behind a Vercel login by default and no one outside your team can open them — this is exactly what happened earlier in this project. If it's still gated, go to Settings → Deployment Protection and turn it off (or set to "Only Preview Deployments").
STEP 5

Connect frontend and backend (CORS)

In your backend's server.js, allow your real Vercel domain — not just localhost:

app.use(cors({ origin: 'https://your-frontend.vercel.app' }))

Commit and push — Render redeploys the backend automatically.

STEP 6

Test the live deployment for real

  1. Open the public Vercel URL in a private/incognito window
  2. Log in with each of the 3 seeded accounts and confirm real data loads
  3. Refresh the page after logging in — you should still be logged in, with the same data
  4. Create a test contract, refresh the page — if it's still there, your database is genuinely persisting, not just sitting in memory
Part 2

Pre-Submission Checklist

Work through this after deployment is live. Click any item to check it off — the progress bar updates as you go.

0 / 0 complete
PHASE 1

Confirm Every Flagged Bug Is Actually Fixed

Log in to the live site for each — don't assume the prompt was followed correctly.

  • Top bar identity bug — top-right profile and sidebar-footer profile show the same name, title, role ("Alex Rivers / Analyst" must not appear anywhere)
  • Sidebar background reaches the bottom of the viewport with no white gap, even on a tall window
  • Every amount shows ₹ with Indian digit grouping (₹12,45,200) — no $ anywhere
  • Settings screen is fully functional — no "Under Development" state remains
  • Manager filter is hidden entirely for employees, on every screen, not just Dashboard
  • All Reports & Analytics charts reflect the same live data — none stuck on mock/hardcoded numbers
  • No stray broken icon/asset glitches on the New Contract Entry form
  • Login helper text is neutral, not "administrator credentials"
PHASE 2

Confirm Core Features Actually Exist

  • A real scheduled job checks renewal dates and writes "Renewal Alert Sent" to the audit log — not just a dashboard count
  • Audit log captures all event types: Contract Created, Price Revised, Category Added — not only Renewed
  • Admin can add/rename equipment categories in Settings, and they appear immediately as chips on the Entry Form
  • Notifications bell shows real alerts, or is removed — never empty/non-functional
PHASE 3

Security Verification

Run these as direct API calls (Postman/dev tools), not just by clicking the UI.

  • Sarah's token cannot fetch a contract belonging to Marcus — must return 403/404
  • Sarah's token cannot call GET/POST on /api/users — must return 403
  • Sarah cannot reassign her own contract to Marcus via a PUT body — manager stays unchanged on refetch
  • Admin cannot deactivate or demote their own account (no self-lockout)
  • Deactivating a manager with active contracts prompts for reassignment, never silently orphans them
  • No API response ever includes password_hash, including GET /api/users
PHASE 4

Cross-Role Functional Walkthrough

  • Admin: Dashboard shows company-wide totals, not scoped to one person
  • Admin: Team Management — add, edit, and toggle-active a test employee
  • Admin: can assign a new contract to any active manager
  • Sarah: Dashboard shows only her contracts, correct non-zero numbers
  • Sarah: new contracts auto-assign to her, manager field locked
  • Sarah: typing /team directly into the URL bar blocks access
  • Marcus: same checks as Sarah — his data is separate from hers
PHASE 5

Device & Visual QA

  • Full walkthrough at 375px, 768px, and 1280px — nothing breaks or overlaps
  • Logo displays correctly on Login and Sidebar at every size
  • All forms show clear inline validation errors for missing required fields
PHASE 6

Production Hardening

  • Demo passwords (admin123 / password) rotated to real, securely-stored credentials before real client data goes in
  • JWT secret, database URL, and any API keys live only in environment variables — never committed to GitHub
  • Deployed URL is the Production Vercel domain — publicly reachable, no Vercel login wall
  • MSA file upload restricted to PDF with a sensible size limit — confirmed it actually rejects other types
  • JWT storage decision finalized (httpOnly cookie recommended) — not left as "either" in the code
PHASE 7

Submission Package

For your reviewer / internship deliverable:

  • GitHub repository URL with a README covering setup steps and the live URL
  • Live deployed URL (public, no login wall)
  • Project report (PDF)
  • Final presentation (PPT)
  • Demo video (4–5 min, shows both Admin and Employee flows)
  • Project logbook / diary entries

For Oxygen Sports (the actual client):

  • A short, plain-language user guide for someone who has never seen the codebase
  • The real (rotated) admin login handed over securely — not over email/WhatsApp in plain text
  • A point of contact for the first couple of weeks after go-live
  • Clear agreement on who owns the Vercel/Render/database accounts going forward
PHASE 8

Final Sign-Off Demo Script

Walk through this exact sequence live, in order, when presenting:

  • Open the public URL fresh — not already logged in
  • Log in as Admin → show Dashboard, Reports, Team Management
  • Add a new employee from Team Management
  • Log out, log in as that new employee → show their empty dashboard
  • Create a new contract as that employee → confirm it auto-assigns to them
  • Log back in as Admin → confirm that contract now appears in the full list
  • Open its Detail & History view → show the audit log entry for its creation
  • Show the Reports page filtering by date range
  • Resize the browser to mobile width → confirm it still works
